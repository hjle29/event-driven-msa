## Critical (반드시 수정)

### C-1. GlobalExceptionHandler가 ErrorCode의 HTTP 상태를 무시 — 모든 BusinessException이 400 반환

- **파일**: `common/src/main/java/com/hjle/common/exception/GlobalExceptionHandler.java`
- **문제**: `@ResponseStatus(HttpStatus.BAD_REQUEST)` 하드코딩으로 인해 `ErrorCode`에 정의된 HTTP 상태(409, 404 등)가 무시되고 항상 400 반환
- **영향**: `SETTLEMENT_ALREADY_COMPLETED`(409), `ENTITY_NOT_FOUND`(404), `HANDLE_ACCESS_DENIED`(403) 등 모두 400으로 응답
- **수정**: `ResponseEntity<ApiResponse<Void>>`로 반환 타입 변경 후 `e.getErrorCode().getStatus()` 사용

### C-2. 단위 테스트 전무

- **파일**: `settlement-service/src/test/` (존재하지 않음)
- **문제**: CLAUDE.md에 "JUnit5, 단위 테스트 필수" 명시되어 있으나 테스트 파일이 없음
- **미커버 대상**:
  - `SettlementService.getSettlementByOrderId` (정상 / 미존재)
  - `SettlementService.getSettlementsByUserId` (정상 / 빈 목록)
  - `SettlementService.completeSettlement` (정상 / 미존재 / 멱등성 가드)
  - `SettlementEntity.completeSettlement`
  - `SettlementResponse.from` (null status 분기)

## Important (수정 권장)

### I-1. `SettlementEntity`가 `dto` 패키지에 위치 — 레이어 컨벤션 위반

- **파일**: `settlement-service/src/main/java/io/github/hjle/settlement/dto/SettlementEntity.java`
- **문제**: JPA `@Entity`는 `domain` 패키지에 위치해야 함 (CLAUDE.md: `controller / service / repository / domain / dto`)
- **수정**: `domain` 패키지로 이동

### I-2. `completeSettlement` 엔드포인트에 접근 제어 없음

- **파일**: `settlement-service/src/main/java/io/github/hjle/settlement/SettlementController.java:27`
- **문제**: 내부 정산 전용 API임에도 인증/인가 없이 누구나 호출 가능
- **수정**: 내부 서비스 시크릿 헤더 검증 또는 Spring Security 역할 제한 적용

### I-3. `SETTLEMENT_ALREADY_COMPLETED` 에러명이 `FAILED` 상태에도 사용됨

- **파일**: `settlement-service/src/main/java/io/github/hjle/settlement/SettlementService.java:75`
- **문제**: `status != READY` 조건에서 `FAILED` 상태도 "already completed" 에러를 던짐 — 의미 불일치
- **수정**: `SETTLEMENT_NOT_IN_READY_STATE` 에러코드 분리 또는 상태별 가드 분리

### I-4. `SettlementResponse.status` 필드가 `String` 타입

- **파일**: `settlement-service/src/main/java/io/github/hjle/settlement/dto/SettlementResponse.java:18`
- **문제**: 엔티티는 `SettlementStatus` enum을 사용하는데 DTO에서 `String`으로 변환 — 타입 안전성 손실
- **수정**: `SettlementStatus` enum 타입 사용

### I-5. Kafka 의존성 제거 여부 확인 필요

- **파일**: `settlement-service/build.gradle.kts`
- **문제**: PR #5(merged)에서 추가한 Kafka 컨슈머가 현재 브랜치에서 삭제됨. 의도적 변경인지 확인 필요

## Suggestions (선택적 개선)

### S-1. `SettlementStatus.PENDING`이 데드코드

어느 코드도 `PENDING`을 할당하지 않음. 비즈니스 의미가 없다면 제거 권장.

### S-2. `@PrePersist` null 체크 불필요

`runSettlement()`에서 빌더로 항상 `READY`를 명시 설정하므로 null 체크는 실행되지 않음.
`@Builder.Default` 사용으로 대체 권장 (프로젝트 메모리 패턴과도 일치).

### S-3. 와일드카드 import 제거

`import org.springframework.web.bind.annotation.*;` → 명시적 import 사용

### S-4. `SETTLEMENT_NOT_FOUND` 도메인 전용 에러코드 추가

현재 `ENTITY_NOT_FOUND(C003)` 사용 중. 도메인 특화 코드 `S002` 추가 권장.
